情绪团子

情绪团子是 Lyriks 里的情绪陪伴型音乐 Agent。它不是单纯的播放器挂件，也不是重度养成游戏，而是在用户听歌、选歌、搜索、分享和停留的过程中，用一个轻量可互动的团子把“我现在想怎么听歌”变成可选择、可反馈、可积累的体验。

核心入口 情绪团子作为全局浮层陪伴入口，承接情绪选择、播放反馈、推荐解释和轻互动。
解决问题 回应音乐产品里常见的情绪入口弱、推荐解释弱、播放陪伴弱、长期记忆弱和社交表达生硬。
产品能力 产品覆盖发现页、排行榜、歌手页、搜索、歌曲详情和底部播放器，并由情绪团子、播放空间与轻量陪伴玩法串联完整体验。
轻游戏机制 以“雨夜补给站”为首个玩法，不设失败、不惩罚、不打断音乐，让情绪在听歌过程中被看见。
产品主张
用情绪团子承接“为什么此刻想听这首歌”，把音乐播放、情绪状态、互动反馈和长期偏好串成连续体验。
EMO 陪伴中

Tab 1 · 产品简介

把“我现在想怎么听歌”做成音乐产品里的明确入口。

传统音乐 App 更擅长围绕歌曲、榜单、歌手和推荐流组织体验，但用户打开 App 的真实动机常常不是找一首确定的歌，而是想找一种状态。情绪团子要做的，就是把这种模糊但高频的听歌动机显性承接下来。

要解决的问题

  • 情绪入口弱：用户想找的是放松、EMO、专注、通勤提神等状态，而不是一首明确的歌。
  • 推荐解释弱：用户常常只能看到“听什么”，却不知道“为什么适合现在”。
  • 播放陪伴弱：传统播放页以封面、歌词和控制器为主，轻互动和情绪反馈不足。
  • 长期记忆弱：每次听歌产生的状态选择和偏好没有沉淀成“越来越懂我”的关系。
  • 社交表达生硬：分享常常只是链接转发，缺少更轻、更有情绪温度的表达。

产品方案

  • 用情绪团子作为全局入口：团子悬浮在音乐 App 中，随播放状态、节奏和用户选择产生反馈。
  • 用一级情绪和二级动作状态组织听歌：例如 EMO、放松、专注、元气，再映射到雨夜补给、海边放松、书桌专注等场景。
  • 用播放空间增强陪听：在 `/play` 页面把当前歌曲、歌词信息、元数据和情绪小游戏放到一起展示。
  • 用轻游戏机制沉淀陪伴感：听歌或点击掉落物即可积累共鸣能量，不设失败，不制造压力。
  • 用长期记忆形成个性化：记录常选情绪、常听风格、偏好时段和打扰偏好，但不做心理诊断。
进入音乐首页 用户先浏览发现页、榜单、搜索或歌曲详情，情绪团子作为全局浮层持续在场。
选择情绪状态 用户主动选择自己此刻想要的听歌状态，系统可推荐，但任何切换都需要确认。
生成推荐方向 系统根据情绪、动作状态、最近听歌行为和场景语境调整推荐基调与歌单旅程。
进入播放空间 当前歌曲封面、歌名、歌手、歌词陪伴文案和元数据在沉浸式听歌空间里共同呈现。
积累陪伴记忆 通过轻游戏反馈、状态选择和歌单偏好，让陪伴关系持续可感知、可积累。

音乐发现与播放

产品保留完整音乐基础能力，包括发现页、榜单页、附近音乐、搜索、歌曲详情和底部播放器，保证情绪能力始终运行在真实音乐主链路之上。

情绪团子浮层

团子承担状态展示、快捷交互和节奏响应三类职责，会根据当前情绪、播放状态和互动模式展示不同表情、动作、皮肤和特效。

情绪状态与设置

状态系统强调克制表达，不诊断用户心理状态，不用强判断语气定义用户，也不强行把低落情绪扭转成开心，而是优先陪伴、承认和轻量引导。

播放空间与核心场景

`/play` 页面是 Lyriks 的沉浸式听歌空间。用户播放歌曲后，可以从旋转封面进入页面，看到当前歌曲封面、歌名和歌手、歌词陪伴文案、专辑/曲风/发行日期等歌曲元信息，以及情绪小游戏“雨夜补给站”。

EMO 放松 专注 元气 平静 低落 孤独
雨夜补给站 EMO 状态下的轻量陪伴玩法，掉落雨滴音符、歌词碎片和共鸣光点，能量满后触发团子状态变化。
海边放松 适配放松和治愈，突出轻晃、呼吸感和慢节奏陪听，强调“陪你待在这个状态里”。
书桌专注 适配专注状态，把书桌、时钟和灵感点映射为陪听场景，让推荐与专注语境一致。
通勤提神 适配元气和平静，结合时间段、最近听歌行为与出行语境提供更具体的推荐解释。

Tab 2 · 产品效果

情绪团子把首页接管、状态切换、推荐解释与播放陪伴组织成完整产品体验。

产品以音乐发现与播放为主线，用情绪团子把“今日该怎么听”“现在适合什么状态”“为什么推荐这首歌”串成连续体验。下面两张效果图对应首页接管建议和状态切换确认两个关键入口。

核心体验模块

用户可以从发现页、榜单页、搜索和歌曲详情进入播放，再由团子承接情绪状态选择、陪听反馈、推荐解释和轻互动，让“听歌”从内容消费升级为被理解、被陪伴的过程。

情绪团子浮层 根据当前情绪、播放状态和互动模式切换表情、动作、皮肤与特效。
播放空间 在 `/play` 页面集中展示歌曲信息、歌词陪伴文案、元数据和情绪小游戏。
雨夜补给站 播放歌曲后自动积累共鸣能量，点击掉落物可以进一步加速，不设置失败和惩罚。
分享表达 围绕“更轻、更有情绪温度”的方向设计团子式分享文案和状态表达。

核心体验原则

  • 不诊断用户心理状态，不输出医学化或武断化结论。
  • 系统可以推荐状态，但任何重要动作都由用户确认后执行。
  • 在低落、EMO、孤独等场景下优先陪伴、承认和轻量引导，而不是强行转正。
  • 轻互动要服务于听歌主线，不打断播放，不把产品重心变成重度游戏。
  • 所有结果优先结构化表达，再由前端组件决定最终 UI 呈现。
体验目标 让用户感知“情绪会影响我此刻怎么听”，而不仅是被动接收推荐流。
体验结果 用户不仅知道“现在听什么”，也能理解“为什么适合现在”以及“接下来可以怎么听”。
音乐基础能力 保留发现页、排行榜、歌手页、搜索、歌曲详情和底部播放器，情绪能力建立在真实音乐信息架构之上。
陪伴入口 团子承担状态展示、快捷交互和节奏响应三类功能，是连接音乐主链路与情绪互动的统一入口。
播放空间 沉浸式 `/play` 页面把当前歌曲、歌词陪伴文案、歌曲元信息和情绪小游戏组织成一页内的连续体验。
轻游戏机制 通过自动积累能量和点击掉落物的方式，让情绪被看见而不是被对抗，延长陪听体验的反馈链路。
推荐表达 不仅告诉用户“听什么”，还通过团子口吻说明“为什么适合现在”以及“接下来可以怎么听”。
长期记忆 记录常选情绪、常听风格、偏好时段和打扰偏好，形成轻量偏好摘要，而非敏感的心理结论。

Tab 3 · 竞品分析

主竞品聚焦汽水音乐、QQ 音乐、网易云音乐。

根据《三种app中陪伴小宠形象调研.docx》的文字描述，三组样本分别代表三种方向：汽水音乐把小宠物接到 AI 音乐发现入口，QQ 音乐把 AI 陪伴做成可选人物、动作和背景玩法，网易云音乐把播放页宠物与自定义生成结合。

汽水音乐：模式选择、小宠物与 AI 音乐发现

  • 调研文档中记录了“多场景模式选择”“浮动小宠物”“点击小宠物跳转 Assistance：AI 音乐发现”。
  • MuChator 论文对应沉浸式信息流发现和交互式音乐推荐，说明自然语言场景找歌已经被产品化验证。
  • 局限是小宠物互动性弱，更像 AI 搜歌助手快捷入口；情绪团子需要让角色本身承担情绪选择和推荐反馈。

QQ 音乐：AI 陪伴、人物选择与背景玩法

  • 调研文档中记录了 AI 陪伴人物选择、人物动作、音乐播放页固定位和背景玩法。
  • 它证明音乐产品里可以承接人物、动作和背景这类视觉资产，适合和会员、装扮体系连接。
  • 局限是音乐页形象位置固定，不可玩、不可交互、不可移动；情绪团子应强调可拖拽、可切换、随音乐变化。

网易云音乐：自定义生成与音乐人格表达

  • 调研文档中记录了播放页宠物和自定义生成宠物形象。
  • 网易云的优势在于音乐情绪、评论社区、音乐人格和状态表达心智，适合做“我的音乐状态”。
  • 局限是宠物主要在固定动作中切换；情绪团子应把生成形象、听歌行为和推荐状态连续绑定。

Tab 4 · 游戏拓展

用“听歌掉落”把陪伴感变成可持续的轻互动。

小游戏采用“轻放置 + 微节奏收集 + 情绪陪伴”的形式：用户不用一直盯着屏幕玩，听歌本身会自动掉落资源；用户想互动时，可以点一点、接一接、抱一抱小宠物，帮它把情绪能量攒起来。这样不会打断音乐 App 的主体验，而是让播放过程多一层陪伴反馈。

产品游戏拓展可以定义为“情绪能量补给站”。用户选择情绪后，小宠物进入对应状态；播放歌曲时，系统根据播放时长、节奏和当前情绪持续积累情绪能量。以 EMO 状态为例，小宠物缩在雨夜角落里，歌曲播放时掉落“歌词碎片”“雨滴音符”“共鸣光点”，用户可以帮它收集这些光点，为它加油。能量满后，小宠物不会立刻“变开心”，而是从“低电量 EMO”变成“被陪伴的 EMO”。重点不是消灭负面情绪，而是承认情绪，陪它待一会儿。

为什么要加入游戏化

  • 情绪设置解决“我现在是什么感觉”，小游戏进一步解决“我怎么陪这个情绪待一会儿”。
  • 播放页原本是工具型空间，宠物和能量条能把听歌过程变成有反馈的陪伴场景。
  • 掉落、收集、解锁动作和装饰，可以自然承接留存、会员权益、皮肤套装和活动运营。

首个玩法：雨夜补给站

  • 用户选择 EMO 或深夜 EMO 后，播放歌曲时进入雨夜场景。
  • 场景上方掉落雨滴音符、歌词碎片、共鸣光点，点击或接住后增加共鸣能量。
  • 能量满后，小宠物从“低电量 EMO”变成“被陪伴的 EMO”，雨变小，小夜灯亮起。

玩法边界

  • 不设置失败、不扣分、不 Game Over，避免给低落情绪用户增加压力。
  • 听歌自动积累能量，用户不操作也能推进；互动只作为加速和情绪表达。
  • 奖励优先做小动作、房间物件、皮肤套装和今日情绪卡，不抢推荐和播放主线。

Tab 5 · 技术方案

以多 Agent 架构为核心，把情绪理解、推荐生成、陪听反馈和长期记忆拆成可控能力。

Lyriks 采用“Agent + toolkit + workflow”的结构。`browser-agent`、`multiModal-agent`、`chat-agent` 负责执行与编排，不同情绪能力则沉淀为可复用的 toolkit，让情绪理解、推荐生成、陪听反馈和长期记忆都具备明确边界。

Agent 层

  • `browser-agent` 负责承接首页、搜索、播放页和状态切换等前台场景，把用户事件、页面语境和播放状态组织成可执行任务。
  • `chat-agent` 负责处理情绪理解、推荐解释、日记生成和分享表达等语言型能力。
  • `multiModal-agent` 负责歌曲高潮、节奏变化、粒子效果和团子动作等多模态反馈。

Toolkit 层

  • 原本以 `emotion` 开头的能力统一收敛为 toolkit，例如 `emotionSelection toolkit`、`playlist toolkit`、`stamina toolkit`、`diary toolkit`、`socialShare toolkit`。
  • `todayHandover toolkit` 负责今日接管建议，`emotionSelection toolkit` 负责识别一级情绪与二级动作状态。
  • `playlist toolkit`、`multimodalEffect toolkit`、`stamina toolkit`、`diary toolkit`、`socialShare toolkit` 分别输出歌单旅程、陪听特效、状态续航、听歌日记和分享表达。

Skill 与 Workflow

  • Workflow 负责把首页接管、情绪切换、歌单推荐、陪听反馈和会话总结归入统一流程，让不同 Agent 能按场景调用合适 toolkit。
  • 结构化输出最终沉淀为气泡、状态、特效、歌单、日记和分享文案，保证 Agent 决策与 UI 呈现可解耦。
  • 所有关键动作都保留用户确认，尤其是切换状态、改变播放队列和发起分享。
场景 Agent / Toolkit 核心输出
首页需要今日歌单推荐建议 `browser-agent` / `todayHandover toolkit` 根据情绪/状态，联网搜索，生成推荐歌单列表。
用户表达“今晚想安静点”或点击情绪设置 `chat-agent` / `emotionSelection toolkit` 一级情绪、二级动作状态和置信度。
用户希望按当前情绪听一组歌 `chat-agent` / `playlist toolkit` 情绪歌单旅程、阶段说明和推荐理由。
歌曲播放到高潮、歌词段落或节奏变化 `multiModal-agent` / `multimodalEffect toolkit` 团子动作、粒子、光效和持续时间。
用户在 EMO、低落或孤独状态停留较久 `chat-agent` / `stamina toolkit` 继续陪伴、温和转场或结束提示。
一段听歌会话结束，需要生成总结或分享表达 `chat-agent` / `diary toolkit`、`socialShare toolkit` 今日听歌日记、状态摘要、分享文案和确认动作。
情绪团子的技术方案不是把 Agent 贴到播放器上，而是让 Agent 成为情绪理解、推荐决策、陪听反馈和长期记忆的中枢。
